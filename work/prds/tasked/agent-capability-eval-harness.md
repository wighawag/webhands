## Resolved decisions

Decisions taken with the user during planning (2026-06-29); recorded so the tasker treats them as settled. (The harness SHAPE, in ## Solution and ## Implementation Decisions, was resolved at launch; these close the agent-under-test wiring, a deterministic self-test fixture, and the per-run account hygiene that were briefly open.)

- **D1: the agent-under-test sits behind a LAUNCH SEAM, modelled on dorfl's `Harness` seam; a GENERIC SHELL/COMMAND adapter is the v1 deliverable and is a fully real capability subject; the pi adapter is a later native-niceties improvement, not needed for the first eval.** The harness defines an `AgentUnderTest` launch seam (an interface: hand it the goal-prompt + the verb-surface reference, it launches a real unaided agent, the harness scores the end state). This mirrors `~/dev/github/wighawag/dorfl`'s proven pattern exactly, including dorfl's `HarnessAdapter = 'null' | 'pi'` split, where the **`null` adapter is the GENERIC SHELL adapter** (`packages/dorfl/src/harness.ts`): it shells out to a configured agent command (`agentCmd`, with a `{model}` placeholder), feeds the prompt on stdin, and captures stdout. It is NOT a stub: it launches whatever REAL agent the command points at, lacking only pi-native conveniences (session file, native liveness; it falls back to PID). The **`pi` adapter** (`pi-harness.ts`) does the SAME job plus those native niceties (`pi --print --session <full-path>`, liveness from the session file rather than PID). We adopt this split:
  - **The generic SHELL/COMMAND adapter is v1 and is sufficient for the first real eval.** It launches a real, UNAIDED agent (any agent invocable as a shell command: `claude -p`, `pi --print`, etc., configured as the agent command), handed ONLY the goal-prompt on stdin + the verb-surface reference. It is a genuine capability subject, subject to the full no-priming rule. The first capability eval runs on THIS adapter; no pi-specific code is required to ship the harness.
  - **The pi adapter is a DEFERRED improvement** (## Out of Scope for v1): the same launch seam with pi-native session/liveness, worth adding for unattended/scheduled runs where a pi session file and native liveness beat shelling-out + PID. It is non-breaking to add later precisely because the launch seam is the durable contract.
  - **The seam is the durable deliverable; the shell adapter is its first (and, for v1, only required) implementation.** Any agent invocable from a shell command plugs in via the command string; pinning a model for comparability is done through the command (dorfl's `{model}` substitution pattern) rather than a pi-only flag.

- **D3: a DETERMINISTIC SCRIPTED RUN is a SEPARATE self-test fixture (the dorfl `NullHarness` analogue), NOT an adapter and NOT a capability subject; it proves the harness MACHINERY without a real agent.** Distinct from the launch adapters in D1 (those launch a real, unaided, non-deterministic agent), this is a fixed, PRIMED sequence of `npx webhands <verb>` calls that the harness machinery runs as if it were an agent's trace, so the harness's OWN logic can be exercised deterministically and for free: contract parsing, the end-state assertion via read verbs, milestone scoring, the pass/fail/INCONCLUSIVE decision, the no-priming guard, and the site-health precheck. Its two jobs:
  1. **Self-test rig:** drive a KNOWN-GOOD scripted trace and assert the harness reports PASS (and the right milestones), and a KNOWN-BAD trace and assert it reports FAIL, so a harness bug is caught without blaming an agent. (Against a LOCAL FIXTURE this part is gate-testable; see ## Testing Decisions.)
  2. **Negative control on a real site:** a deliberately primed, scripted end-to-end run that confirms the end-state-assertion machinery actually fires against the live target, isolating "is the harness correct?" from "is the agent capable?".
  CRUCIAL DISTINCTION the tasks MUST preserve: the scripted run is PRIMED by construction, so a green scripted run is NEVER a capability pass. The no-priming rule (## Implementation Decisions) binds only the D1 capability-subject adapters, never this fixture. Keeping it SEPARATE from the shell adapter is the point: the shell adapter launches a real agent (a capability subject); this fixture replays a fixed script (a machinery check). Conflating them would let a primed script masquerade as a capability result.

- **D2: per-run account hygiene. A FRESH uniquely-tagged account each run is the CORRECTNESS mechanism; a post-verify, best-effort DELETE is tidiness, never the success check. Assert FIRST, delete SECOND, and a non-pass run KEEPS the account for inspection.** Layered, in this strict order:
  1. **Create a fresh, per-run-NONCE-tagged identity every run** (e.g. `webhands-eval-<runId>@…`, and where the artifact is a value, tag it too: a transfer amount/memo carrying the nonce). This is what makes runs INDEPENDENT and the end-state assertion UNAMBIGUOUS: the harness asserts against THIS run's artifact, never a leftover from a prior run. Correctness does not depend on any cleanup succeeding.
  2. **Make the end-state assertion (and milestone checks) BEFORE any cleanup.** Deletion is teardown, never part of the verdict.
  3. **Then best-effort DELETE the account where the site supports it** (AutomationExercise has an in-flow delete-account step; ParaBank has no clean delete, so rely on the nonce-tagged artifact there). A failed delete (or a site with no delete) NEVER flips the verdict; the fresh-random-account already guaranteed correctness, so delete only keeps the sandbox from accumulating cruft.
  4. **A FAIL or INCONCLUSIVE run does NOT delete.** It leaves the account/state for inspection (destroying evidence on failure is the wrong default). Only a clean PASS triggers best-effort cleanup.
  - Per-tier note: SauceDemo needs none of this (reset-free, fixed demo credentials). The fresh-account + assert-then-delete discipline is the Tier-2 (AutomationExercise / ParaBank) contract. Honest caveat carried into those tasks: a site's signup may rate-limit repeated fresh accounts. If so, that surfaces as INCONCLUSIVE (a harness/environment limit), not a capability fail, and is a signal to widen the run interval or seek a resettable instance.

## Problem Statement

The repo just shipped a broadened agent VERB SURFACE (`broaden-agent-verb-surface`, now in `work/prds/tasked/`, 8 tasks merged): `navigate`, `snapshot`, `query` (+ state shorthands `exists`/`count`/`isVisible`/`getAttribute`), `click`, `type`, `press`, `hover`, `select`, `scroll`, `drag`, the coordinate `mouse`, `screenshot` (returns a file PATH), frame-scoped `eval` (`--frame`), durable `query` refs, `wait`, `cookies`. Every verb is proven by a DETERMINISTIC LOCAL-FIXTURE test that exercises it in isolation. Those fixture tests prove each verb works MECHANICALLY on a clean DOM we control.

But mechanical-correctness-in-isolation is not the thing the project actually cares about. webhands' north star is that a capable agent, given ONLY the verb surface and a goal, can drive ANY site end to end, the way a person could by hand, including self-solving a captcha with its own key (both captcha families are already proven verbs-only against local fixtures: `work/notes/findings/click-and-type-already-frame-scoped-via-framelocator.md`, and the two captcha-proof tasks in `tasks/done/`). Nothing in the repo MEASURES progress toward that north star on REAL sites. There are two specific blind spots the local fixtures cannot cover:

- **Composition.** A verb that passes its isolated fixture test says nothing about whether an agent can COMPOSE the verbs to accomplish a real, multi-step job (register, open accounts, transfer funds, confirm) on a site it has never seen and was given no script for.
- **Messy-real DOM regressions.** A verb can work perfectly on a clean fixture and FAIL on a real, messy, framework-reconciling production-like DOM (the exact gap a local fixture is structurally unable to reveal). The agent-facing surface can regress in ways the gate is blind to.

The manual Kayak smoke (`docs-tos-humility-and-kayak-smoke`, `tasks/done/`) is the closest prior art, and it is deliberately "live and flaky by nature, NOT a CI gate." It proves the PIPE end to end but it is a hand-run snapshot read, not a capability measurement, and not repeatable as a pass/fail scoreboard.

What is missing is a CAPABILITY SCOREBOARD: a repeatable, opt-in harness that hands an unaided agent a high-level natural-language goal on a real, unfamiliar, multi-step site and mechanically checks whether the agent reached the goal. It is deliberately SEPARATE from the deterministic correctness gate, because a real third-party site rots, rate-limits, and goes down, and must NEVER red the build.

## Solution

A test harness whose unit is an **eval**: a pair `{goal-prompt, end-state-assertion}` run against a real (preferably sandbox) site.

- **goal-prompt:** a high-level natural-language goal handed to an unaided agent-under-test, e.g. "register an account, then transfer $500 between two of your accounts and confirm it" or "create an account". The prompt is precise enough that success is MECHANICALLY verifiable, but it contains NO selectors, NO step list, NO site-specific foreknowledge. The agent gets the goal + the webhands verb surface and nothing else. (See the no-priming rule, a first-class property below.)
- **end-state-assertion:** a DETERMINISTIC check, run by the HARNESS (not the agent), that passes/fails on the resulting state: the transaction row appears, the account exists / the logged-in state is reached, the order id appears, the cart count is N.

The agent's PATH is free (it discovers the site live); only the END STATE is checked. That is how the harness stays deterministic despite a non-deterministic agent.

Four load-bearing properties, each a hard constraint from this codebase's conventions:

1. **NON-GATING by construction.** These evals are OPT-IN and live OUTSIDE the deterministic verify gate (`pnpm format:check && pnpm build && pnpm test`), which runs against LOCAL FIXTURES only. The evals live in their own directory behind their own command, EXPLICITLY excluded from `pnpm test`, run manually or on a schedule, never in `verify`. A flaky external site must NEVER red the build. This is the same "manual by nature" stance the Kayak smoke and the iamhuman example already document.

2. **DETERMINISTIC EVALUATION despite a non-deterministic agent.** The harness itself makes the end-state assertion using webhands' OWN read verbs (`query`/`snapshot`/`exists`/`count`/`getAttribute`) against the live page AFTER the agent reports done, optionally corroborated by the site's own confirmation surface (a confirmation page / order id). The assertion does NOT trust the agent's self-report; the agent's "I'm done" only TRIGGERS the harness's independent verb-based check. (Decision recorded in ## Implementation Decisions; the alternative, trust-self-report-plus-spot-check, is rejected there.)

3. **NO HARDCODED FOREKNOWLEDGE given to the agent.** This is what makes it a CAPABILITY eval and not a scripted test. The agent receives the goal-prompt + the verb surface; it never receives selectors, step sequences, or "this site's login is at /login". The no-priming rule is a first-class, asserted property of the harness, not a guideline.

4. **SAFE, REPEATABLE, AUTHORIZED targets.** Prefer realistic SANDBOXES built to be automated over production sites (production carries anti-bot / ToS / 2FA / real-state hazards, per ADR-0002 and the README humility note). Targets are TIERED by difficulty and each is assessed for fit and per-run state hygiene (see ## Implementation Decisions).

The harness drives webhands through its existing surface (the `npx webhands <verb>` CLI path the README leads with, and/or the long-lived `serve` session, ADR-0005), against a warmed profile, honoring the stealth/proxy options that already exist (ADR-0009). It adds NO new verbs: the surface is complete for this. If running a real eval reveals a missing or messy-DOM-broken verb, that is a FINDING (and possibly a future surface PRD), captured per the work/ contract, NOT new scope here.

## User Stories

1. As a maintainer, I want a repeatable harness that hands an unaided agent a high-level goal on a real site and tells me pass/fail, so I can MEASURE webhands' capability on real sites, not just mechanical verb correctness on fixtures.
2. As a maintainer, I want each eval expressed as `{goal-prompt, end-state-assertion}`, so the success criterion is explicit, mechanical, and reviewable.
3. As a maintainer, I want the end-state assertion made by the HARNESS using webhands' own read verbs (and/or the site's confirmation surface), NOT by trusting the agent's self-report, so a hallucinated "done" cannot pass an eval.
4. As a maintainer, I want the agent-under-test to receive ONLY the goal-prompt + the verb surface (no selectors, no steps, no site foreknowledge), so the harness measures genuine unaided capability, not a script.
5. As a maintainer, I want these evals to live OUTSIDE the `verify` gate, in their own directory behind their own command, explicitly excluded from `pnpm test`, so a flaky third-party site can never red the build.
6. As a maintainer, I want a Tier-1 SauceDemo eval (login + sort + add-to-cart + multi-step checkout) as the simplest, most stable proof, including a goal whose success requires the agent to DISCOVER special-user behaviour (e.g. `problem_user` / `performance_glitch_user`) rather than be told it.
7. As a maintainer, I want a Tier-2 stateful/branching eval (AutomationExercise full e-commerce: signup, login, cart, address, payment, order; or ParaBank register, open accounts, transfer funds, confirm transaction), so the scoreboard covers a consequential, strongly-stateful flow.
8. As a maintainer, I want a Tier-3 messy-real eval (Magento demo's production-like DOM), so the harness catches the "works on a clean fixture, fails on a real messy DOM" regression that local fixtures structurally cannot reveal.
9. As a maintainer, I want each eval to score with MILESTONE partial credit (e.g. reached login / reached cart / completed checkout) in addition to a final binary pass/fail, so a near-miss is a useful capability signal and not just "fail".
10. As a maintainer, I want the harness to DISTINGUISH "the agent failed" from "the site was down / rate-limited / changed", so an external-site outage is reported as INCONCLUSIVE, never as a capability regression.
11. As a maintainer, I want per-run account/state hygiene handled per target (fresh account each run where the site allows, in-flow cleanup, or per-run-unique artifacts), so re-running an eval does not wreck state or make the assertion ambiguous.
12. As a maintainer, I want the harness to drive webhands through its EXISTING surface (`npx webhands <verb>` CLI and/or the `serve` session, ADR-0005) against a warmed profile honoring the stealth/proxy options, so it exercises the real agent-facing path, not a private back door.
13. As a maintainer, I want the harness to add NO new verbs; if an eval reveals a missing or messy-DOM-broken verb, I want that captured as a FINDING for a future surface PRD, so this harness stays a measurement tool, not a surface change.
14. As a maintainer, I want WebArena noted as a FUTURE full self-hosted benchmark suite, out of initial scope, so the ambition is recorded without bloating v1.
15. As a maintainer, I want the harness's non-gating-ness to be ENFORCED (a separate command/dir, demonstrably not invoked by `pnpm test`), not just documented, so it cannot silently creep into the gate later.
16. As an agent-under-test, I want my goal handed to me as plain natural language (on stdin) and my verb access wired through one clear channel (the bash/CLI `npx webhands <verb>` path), so any agent invocable as a shell command can be pointed at an eval without bespoke glue.
17. As a maintainer, I want a clear catalogue of evals (each with its tier, target, goal-prompt, end-state-assertion, and milestone set), so the scoreboard is legible and a new eval is cheap to add.

## Out of Scope

- **Adding any new verb.** The surface is complete for this harness. A missing or messy-DOM-broken verb discovered while running an eval is a FINDING (captured per the work/ contract) and possibly a future surface PRD, NOT scope here.
- **Making these evals GATE CI.** They are opt-in and non-gating by nature; they must never enter the `verify` gate (`pnpm format:check && pnpm build && pnpm test`). The harness's own unit logic may be gate-tested (above); the live-site evals never are.
- **A pi-NATIVE adapter.** v1 ships the generic shell/command adapter (D1), which launches a real unaided agent (including pi, invoked as `pi --print`) and is a sufficient capability subject. A pi-native adapter (pi session file + native liveness, the upgrade over shell-out + PID) is a DEFERRED improvement for unattended/scheduled runs, addable later through the same launch seam without breakage. Not needed for the first eval.
- **WebArena full-benchmark integration.** Noted as a FUTURE self-hosted benchmark suite; out of initial scope.
- **Driving PRODUCTION sites as standing evals.** Production carries anti-bot / ToS / 2FA / real-state hazards (ADR-0002, README humility note). Sandboxes built for automation are preferred; a production target, if ever added, is a deliberate, separate, human-gated decision.
- **A built-in captcha solver or provider key.** Unchanged from the surface PRD: webhands ships neither. An eval that involves a captcha relies on the agent-under-test bringing its own key, exactly as the surface PRD frames it.

## Further Notes

- **The WHY (north star).** webhands' north star is that a capable agent, given only the verb surface, can drive ANY site, including handling captchas with its own key (both families already proven verbs-only against local fixtures). This harness is how we MEASURE progress toward that north star on REAL sites, and how we catch regressions in the agent-facing surface that local fixtures cannot reveal (a verb that works on a clean fixture but not a messy real DOM). It is the capability SCOREBOARD, deliberately separate from the deterministic correctness GATE.
- **Complement, not replacement.** The shipped local-fixture verb tests prove the verbs work MECHANICALLY in isolation; this harness proves the next thing up: that an agent can COMPOSE them to accomplish real jobs on real, unfamiliar sites. Both matter; they answer different questions.
