# Cross-invocation session persistence: `incur serve` hosts the one long-lived browser; verbs are thin clients

A single browser session is kept alive between separate CLI invocations by a long-lived **`incur serve`** process (`cli.fetch` over `/mcp`/HTTP) that owns the Playwright browser and the one active `Session`/page; each `webhands <verb>` process is a **thin client** that calls the running server, which drives its already-live page, and exits. The browser is launched ONCE in the served process, never per verb invocation. We chose this over a bespoke background daemon because the `incur` server is already being built for the CLI/MCP surface (stories 12-14), so the long-lived host and its transport come for free with no second process model or hand-rolled IPC to design, test, and package; and because it lands ADR-0001 literally - the served process IS the controller that owns the long-lived control loop. A separate daemon would only earn its keep if the browser host had to be independent of the incur HTTP server, which v1 does not need (YAGNI).

## Consequences

- **Lifecycle is EXPLICIT (no auto-spawn in v1).** The user/agent runs `serve` (and `launch`/`attach`) to bring up the session; a verb invocation that finds **no live server** prints a clear, actionable error naming the fix (run `serve` first) and exits non-zero - it does NOT silently spawn a browser. This matches the project's house style (explicit, least-magic, clear errors - cf. the binary-install and attach decisions). Auto-spawn-on-first-verb is a deliberate future enhancement, deferred for its race/ownership complexity.
- **Discovery via an endpoint file.** The served process writes its endpoint (port/socket path) under the config dir (`~/.webhands/`); client verbs read it to find the server. This is a SHARED-WRITE location, so tests MUST isolate the config dir (and the profile dir) to a temp dir and assert the real one is untouched.
- **Single session in v1.** The served process holds EXACTLY ONE active session; a second `launch`/`attach` while one is live is a clear "a session is already active; stop it first" error, not a second browser. Multi-session orchestration stays out of scope.
- **Teardown** is explicit (`stop`, or the served process exits on signal), closing the browser.
- **Test seam:** start the served process against an isolated temp config dir, then run two separate client invocations against the LOCAL FIXTURE PAGE and assert the second reuses the first's live session (state set via call 1 is observed via call 2). The on-disk profile persists login across server restarts; this ADR is about persisting the LIVE session (navigation/page state) across client invocations, which the on-disk profile alone does not do.
- **Coupling accepted:** session persistence is tied to the incur server running. Verbs driving a browser with no HTTP server in the picture would need the rejected daemon path; not a v1 need.
